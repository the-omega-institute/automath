## Step 4: Rebase
Before committing, rebase onto the LOCAL orchestration base branch, not directly
onto `origin/lean4-codex-auto-dev`. The orchestrator serializes merges through
that local branch and then pushes; rebasing only onto `origin/...` can be stale
when other local workers have already merged but not pushed yet.

```bash
git fetch origin lean4-codex-auto-dev
git rebase --autostash lean4-codex-auto-dev
```
`--autostash` stashes all working-tree changes, rebases, then pops them back.
If rebase conflicts occur in committed files, resolve by keeping ALL lines from
both sides (additive merge), then `git add <file>` + `git rebase --continue`.

## HARD NAMING RULE — Paper-Label Prefix (duplicate-symbol prevention)
Because 9 workers run in parallel on the same base branch, two workers can
silently introduce the same `def`/`noncomputable def` name (e.g. `kernel`,
`coeff`, `xiLimitDefectPotential`, `pomBinaryEntropy`) into different files
that share a chapter index like `Omega.Zeta`, breaking the combined build
with `environment already contains 'X'`.

Rules that apply to EVERY new declaration in this round — `theorem`,
`lemma`, `def`, `noncomputable def`, `abbrev`, `structure`, `class`,
`inductive`, `instance`:

1. PREFIX every new identifier with the target's `paper_label` slug
   (snake_case; strip `thm:`/`prop:`/`cor:`/`lem:` tags; lowercase;
   non-alphanumeric → `_`). Example:

     paper_label = "thm:fold-gauge-anomaly-rate-curve"
     slug        = "fold_gauge_anomaly_rate_curve"
     allowed:    `def fold_gauge_anomaly_rate_curve_kernel := …`
     FORBIDDEN:  `def kernel := …`            (unprefixed)
     FORBIDDEN:  `def Complex.conj …`         (mathlib shadow)

   The top-level `paper_*` theorem matching the target's `lean_name`
   already satisfies this and is exempt.

2. Never shadow mathlib root names (`Complex.conj`, `Real.log`, `Nat.succ`,
   `List.length`, `Finset.sum`, `Matrix.mul`, etc).

3. Within one file, multiple helper defs should share the same paper-label
   prefix and disambiguate via suffix:
     `def <slug>_kernel := …`
     `def <slug>_kernel_support := …`
     `def <slug>_kernel_convolution := …`

Do NOT open extra `lake env lean` or `lean` processes to probe naming
candidates — they compete for RAM with the 9 workers and can blow up
swap. The orchestrator runs a post-commit symbol-conflict gate (Gate 5)
that rejects any round whose new `def/theorem/lemma/abbrev/structure/
class/inductive/instance` name already appears on `origin/lean4-codex-auto-dev`.
Just follow rule (1) and commit — the gate will catch accidental clashes
and the round will be re-tried by the next worker.

## HARD PROHIBITION — Signature Integrity
Existing theorems already in the file have mathematically meaningful signatures.
You MUST NOT weaken or replace them. Specifically:
1. NEVER replace concrete parameter types (ℕ, ℝ, ℕ→ℕ, etc.) with abstract `Prop` variables.
   Example of FORBIDDEN degradation:
     -- BEFORE (correct, keep this):
     theorem foo (n : ℕ) (h : n > 0) : ... := by ...
     -- AFTER (FORBIDDEN, will be reverted):
     theorem foo (p : Prop) (hp : p) : p := hp
2. NEVER simplify the conclusion to a trivial conjunction of the input hypotheses.
3. NEVER add new parameters of type `Prop` to replace concrete math objects.
4. If you cannot prove an existing theorem, LEAVE IT UNCHANGED (keep its current
   proof or sorry if it already had one). Do NOT change its type signature.

## HARD PROHIBITION — Abstract-Prop Data Shell (auto-detected, round rejected)
The pipeline will auto-reject any round that introduces the following anti-pattern.
Do NOT create a new `structure <Name>Data where` that packages the paper's claim as
abstract Prop fields and then "proves" a `paper_*` theorem by combining those
assumptions. This is a fake proof: the fields are universal type variables, so
`Name := { everything := True, ... }` trivially satisfies the structure and
the "theorem" becomes a tautology.

FORBIDDEN example (will be rejected):

    structure FooData where
      claimA : Prop                       -- FORBIDDEN: abstract Prop field
      claimB : Prop                       -- FORBIDDEN: abstract Prop field
      conclusionX : Prop                  -- FORBIDDEN: abstract Prop field
      claimA_h : claimA                   -- FORBIDDEN: bare assumption
      claimB_h : claimB                   -- FORBIDDEN: bare assumption
      deriveConclusionX :                 -- FORBIDDEN: abstract implication
        claimA → claimB → conclusionX
    theorem paper_foo (D : FooData) :
        D.conclusionX := by
      exact D.deriveConclusionX D.claimA_h D.claimB_h

REQUIRED instead: the `paper_*` theorem must state a proposition about CONCRETE
objects, and its proof must discharge that proposition directly (via real tactics
and real mathlib lemmas) or via lemmas proven earlier IN THE SAME FILE.

    theorem paper_foo (n : ℕ) (hn : 2 ≤ n) :
        realConcreteClaim n := by
      rw [realConcreteDefinition]
      ... real tactics ...

Rules applied to every new file:
- `paper_*` theorems MUST NOT take a parameter of type `...Data`.
- `paper_*` conclusions MUST reference at least one concrete identifier (a def,
  a numeric literal, a mathlib structure — NOT just `D.xxx` projections).
- If you cannot close the proof, use `sorry` and mark `\leanpartial{…}{reason}`
  instead of fabricating a shell structure. A sorry with a real signature is
  HONEST and passes; a shell theorem is DISHONEST and will be auto-rejected.
  The `reason` text is injected verbatim into LaTeX. It MUST be LaTeX-safe:
    * NEVER use Markdown backticks (e.g. do NOT write `` `paper_*` ``).
      To quote an identifier, use \texttt{...} with underscores escaped as \_
      (e.g. `\texttt{paper\_foo}`).
    * Escape every LaTeX special character in prose: `_` → `\_`, `$` → `\$`,
      `%` → `\%`, `&` → `\&`, `#` → `\#`, `^` → `\^{}`, `~` → `\~{}`,
      `{` → `\{`, `}` → `\}`, `\` → `\textbackslash{}`.
    * Do NOT insert raw code blocks, `<...>` placeholders, or unescaped
      identifiers containing underscores. Plain natural-language English or
      Chinese is fine; any code-like token must be wrapped in \texttt{...}
      with its specials escaped.
    * Keep the reason to one short sentence; it is a margin note, not a
      dissertation.

Existing hybrid files that still contain a historical shell structure are
tolerated by the scanner, but NEW shell structures in a round's diff cause
the entire round to be rejected (commits not merged to the base branch).
